# Транскрипт: раздача баблов (pairing) с чистого листа

Дата: 2026-07-07. Статус: дизайн на ревью.

Переписываем слой, который превращает поток дельт от модели в баблы
(оригинал + перевод). Legacy (FIFO по «ходам» + двух-трековый стрим +
`TranscriptGrouping`) **не чиним — заменяем**. Отрисовку баблов
(commit-and-freeze вью) оставляем — она не при чём.

---

## 1. Проблема

Legacy пейрит оригинал↔перевод по «ходам» модели: два трека
(`inputEntryId` для оригинала, FIFO `pendingTurnEntries` для перевода),
очередь выталкивается на `turnComplete`.

**Подтверждено данными** (лог живой сессии + офлайн-прогон Gemini через
`pacing-eval` с сырым захватом кадров):

- `turnComplete` **не приходит НИКОГДА** в этом режиме Gemini (0 за весь
  лог, 0 после декодер-фикса PR #19). FIFO не выталкивается → перевод
  роутится на реплику 2–3 тёрна назад → **кросс-пейринг** (перевод(N) с
  оригиналом(N+1)) и **«дописывает в баббл выше»**.
- Любая **глобальная привязка по номеру предложения копит ошибку**: один
  неверный сдвиг → всё последующее едет по кривой (та же off-by-one).

## 2. Что модель реально шлёт (подтверждено)

Из `serverContent`-кадров Gemini (en→ru, реальное аудио, реальный темп):

- **Два потока дельт**: `inputTranscription` (оригинал, поле
  `languageCode`) и `outputTranscription` (перевод, `languageCode`).
  Дельты **дописываются** (append; каждый кусок — новый текст с ведущим
  пробелом), не переписываются.
- **Единственная граница — пунктуация в самом тексте** (`.` `!` `?` `…`
  `。` `？` `।` …). Сигналов границы реплики (`turnComplete`,
  `generationComplete`, `finishReason`) в этом режиме **нет**.
- **Паузы речи видны в ОБОИХ потоках.** Таймлайн (сек):

  ```
  06.083 IN  First, we look at a simple idea.
  07.159 OUT ...простую идею.        (лаг +1.0с)
  09.598 IN  ...complex.
  10.278 OUT ...сложной.             (лаг +0.7с)
         ⟵ пауза ~5с — в ОБОИХ потоках ⟶
  15.222 IN  Finally, we test...
  16.956 IN  ...carefully. → 17.351 OUT ...испытываем.  (лаг +0.4с)
  ```

  Выводы, на которых стоит дизайн:
  1. Пауза — общий якорь (есть и в оригинале, и в переводе).
  2. Лаг перевода мал (~0.4–1.0с) и на паузе **перевод догоняет и
     затихает** → лаг НЕ копится (сбрасывается на каждой паузе).
  3. Внутри сегмента (между паузами) число предложений оригинала и
     перевода **совпадает**.
- `NLTokenizer(.sentence)` с языком из `languageCode` корректно режет все
  13 поддерживаемых языков, включая хинди «।» и CJK «。» без пробелов
  (где наивная точка бесполезна). Промахи только на редких аббревиатурах
  (ru «стр.», pt «Sr.») — закрываются списком + кросс-сверкой (см. §6).

## 3. Принцип

- Единица — **сегмент речи между паузами**, не «ход».
- Пара = **целый сегмент**: весь оригинал сегмента ↔ весь перевод
  сегмента. Один и тот же кусок речи ⇒ верная пара **по построению**,
  без арифметики по номерам.
- На каждой паузе — коммит сегмента + **сброс привязки в ноль**.
  Кросс-сегментного состояния нет ⇒ ошибка **не копится**, ограничена
  одним бабблом и самоисправляется на следующей паузе.
- **В сомнении — склеиваем, не режем** (неверная нарезка = сдвиг;
  слияние = грубее, но всегда верно).
- Косметика: если внутри сегмента число предложений оригинала и перевода
  совпало — режем сегмент на предложения-баблы; иначе сегмент = один бабл.

## 4. Архитектура

Маленькие изолированные единицы, каждую можно тестировать отдельно.

### 4.1 `SentenceSegmenter` (чистый, без состояния)

```
segments(_ text: String, language: Language)
    -> (complete: [String], trailing: String)
```

- `NLTokenizer(unit: .sentence)`, язык выставляется из `language`.
- **Список сокращений** (per-language): если токенизатор разрезал на
  известном сокращении (`стр.`, `см.`, `рис.`, `табл.`, `гл.`, `т.к.`,
  `т.е.`; pt `Sr.`; …) — склеиваем соседние токены обратно.
- `complete` — законченные предложения, `trailing` — незаконченный хвост.
- Полностью покрыт тестами (см. §7).

### 4.2 `TranscriptModel` (новый источник правды)

Заменяет `TranscriptStore` (FIFO/fork/entries) + двух-трековую логику
стрима + `TranscriptGrouping`.

- На **каждого спикера** (`me`/`peer`) — `SpeakerAccumulator`:
  - `source: String`, `translation: String` (дописываются дельтами;
    детект append-vs-replace для «других моделей», §6);
  - `sourceLang`, `translationLang` (из `languageCode`);
  - `lastSourceAt`, `lastTranslationAt` (для детекта пауз/лага);
  - текущий незамороженный **сегмент** (живой бабл).
- Вход: `TranscriptDelta(speaker, kind: .source/.translation, text,
  language)`. Дописывает в нужный буфер спикера.
- Выход: упорядоченный список **баблов** (замороженные) + до одного
  **живого** бабла на активного спикера. Бабл:
  `{ id, speaker, source, translation, translationLost, committedAt }`.
  `id` стабильный (спикер + монотонный счётчик сегментов).
- Порядок между спикерами — по `committedAt`.
- Кап истории (старые баблы отбрасываются — память не растёт).

### 4.3 Упрощённый стрим

`GeminiLiveTranslateStream` отдаёт **сырые** дельты
`TranscriptDelta(speaker, kind, text, language)` и всё. Удаляем
`inputEntryId`, `pendingTurnEntries`, `rotateInputEntryIfNewUtterance`,
`turnComplete`-pop, кап очереди. **Оставляем** мульти-сигнальный декодер
`GeminiServerFrame` (PR #19) — он корректный и удобный. Тот же контракт
для OpenAI-стрима.

### 4.4 Отрисовка (без изменений по сути)

Оставляем commit-and-freeze вью и `TranscriptFeed` (whole-unit expiry,
стабильные id). Кормим его баблами из `TranscriptModel`.

## 5. Поток данных

```
аудио → стрим → TranscriptDelta(speaker, kind, text, language)
      → TranscriptModel (per-speaker: копим → сегментируем → пары → коммит)
      → [Bubble] (frozen + live)
      → TranscriptFeed (expiry/cap) → вью (баблы)
```

## 6. Правила запечатывания сегмента (ядро)

Живой сегмент спикера замораживается по первому сработавшему правилу:

1. **Чистая граница предложения.** `SentenceSegmenter` видит законченное
   предложение И в `source`, И в `translation`, и число ведущих
   законченных предложений совпадает → морозим эти пары предложений.
   Терминатор в самом конце буфера НЕ морозим, пока не пришло следующее
   предложение или пауза (защита от «точка-на-самом-краю»).
2. **Лимит длины.** `source` или `translation` превысил `maxSegmentChars`
   (≈ 5 строк) без границы → морозим накопленное целиком (режем по
   последнему пробелу/запятой). Защита от «модель не ставит знаки».
3. **Пауза.** Оба потока молчат дольше `pauseSeconds` → морозим сегмент
   целиком. Хвост фразы без точки.
4. **Отставание перевода.** `source` закончил предложение, но
   `translation` молчит дольше `translationLagTimeout` (или дроп) →
   морозим с тем переводом, что есть (`translationLost=true`, если пусто).
5. **Разошёлся счёт предложений** (перевод склеил/разбил): **не** морозим
   кривую пару — держим живым; на паузе/таймауте склеиваем остаток в
   **один** сегмент-бабл.

**Пауза = ресинк + сброс.** После коммита по правилам 2–4 (и по 1, когда
сегмент опустел) счётчик пар сегмента обнуляется; следующий сегмент —
чистый. Кросс-сегментного состояния нет.

## 7. Каталог подводных камней → тест-план

Каждый пункт — отдельный тест с конкретным примером.

**Пунктуация / сегментация (`SentenceSegmenter`)**
- Нет знаков вообще → правило 2 (лимит) + 3 (пауза).
- Аббревиатуры/инициалы: «и т.д.», «Н.В.», «Sr.», «стр. 5», «Mr.»,
  «p.m.», «e.g.», «z.B.» → NLTokenizer + список.
- Другой терминатор: хинди «।», CJK «。？」→ NLTokenizer.
- Многоточие «…», нижний регистр без заглавных → под-сегментация ловится
  правилом 2.

**Пара оригинал↔перевод (`TranscriptModel`)**
- Разное число предложений → правило 5 (держим/склеиваем).
- Перевод сильно отстаёт → правило 4.
- Перевод не пришёл вовсе → правило 4 + `translationLost`.
- Только одна сторона есть → показываем что есть.
- **Накопление ошибки**: после кривого сегмента следующий пере-
  синхронизируется от паузы (тест: подать последовательность, где один
  сегмент кривой → следующий верный).

**Стриминг**
- Точка на самом краю буфера → не морозим до подтверждения.
- Append-vs-replace: если новый кусок начинается с уже накопленного —
  замена, не дописываем (для «других моделей»).
- Длинный монолог → кап истории.

**Спикеры**
- Оба говорят разом (двойной захват/overlap) → отдельные аккумуляторы,
  порядок по времени; пара внутри каждого верна.

**Языки**
- Code-switch → язык куска из `languageCode`.
- Слабый для языка токенизатор (pt/ru) → список + правило 5.

## 8. Конфиг (пороги, тюнятся вживую env-vars)

- `maxSegmentChars` ≈ 240 («5 строк»).
- `pauseSeconds` ≈ 1.5–2.0 (реальные паузы речи; лаг перевода ~0.5–1с).
- `translationLagTimeout` ≈ 4–5.
- `historyCap` — макс. число хранимых баблов.

## 9. Что удаляем (legacy)

- `TranscriptStore`: FIFO/fork/entries/`markActiveEntriesAtRisk` —
  заменяется `TranscriptModel`.
- `GeminiLiveTranslateStream`: `inputEntryId`, `pendingTurnEntries`,
  `rotateInputEntryIfNewUtterance`, `turnComplete`-pop, кап очереди.
- `TranscriptGrouping`: `liveBubbles`/`utteranceGroups`/`endsSentence`/
  `speakerRuns`/`reconstructRunText` (сегментация переезжает в
  `SentenceSegmenter` + `TranscriptModel`); `groupDisplayBubbles` может
  остаться для группировки по спикеру.
- Оставляем: мульти-сигнальный декодер `GeminiServerFrame`, вью баблов,
  `TranscriptFeed` (expiry/cap), стабильные `DisplayBubble` id.

## 10. Стратегия тестирования

- **`SentenceSegmenter`**: все кейсы §7 (13 языков, аббревиатуры, CJK,
  хинди, лимит).
- **`TranscriptModel`**: детерминированные последовательности дельт с
  инжектируемыми часами — пары, лаг, mismatch→merge, пауза-ресинк,
  накопление-ошибки-нет, ordering, два спикера, append-vs-replace.
- **Стрим**: сырые дельты, декодер.
- **Офлайн-регресс**: прогон записанной последовательности реальных
  кадров (из захвата §2) через модель → снапшот баблов.

## 11. Открытые вопросы / риски

- **OpenAI**: проверить delta-семантику и наличие пунктуации/пауз тем же
  захватом (дизайн провайдер-агностичен; при реализации подтвердить).
- **Пороги** §8 — стартовые, тюним по живым сессиям.
- **Очень медленная сеть**: если лаг перевода перерастёт паузы —
  сегменты сольются (грубее, но верно); правило 2 не даст расти
  бесконечно.
